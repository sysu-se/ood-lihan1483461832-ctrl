# EVOLUTION.md

## 1. 你如何实现提示功能？

本次作业中的提示功能主要分成两个层次：候选数提示和下一步提示。

候选数提示由 `Sudoku` 对象负责计算。`Sudoku` 保存当前 9x9 棋盘和固定格信息，当用户选中一个空白格并点击“候选”时，程序会调用 `getCandidates(row, col)`。这个方法会从 1 到 9 逐个尝试数字，并检查该数字是否和当前行、当前列、当前 3x3 宫里的已有数字冲突。如果不冲突，就把这个数字加入候选列表。这样得到的候选数不是最终答案，而是根据当前盘面规则可以填写的数字集合。

下一步提示由 `findNextHint()` 实现。它会扫描整个棋盘的空格，优先寻找只有一个候选数的格子。如果某个格子只有一个候选数，说明这一步比较确定，可以提示玩家直接关注这个格子。如果当前没有唯一候选格，程序会寻找候选数最少的格子，把它作为下一步探索的建议位置。这样提示功能不是简单地直接给答案，而是帮助玩家理解当前局面中最值得分析的位置。

在 UI 层，`grid.js` 中的 `showCandidates()` 和 `showNextHint()` 负责把领域层算出的结果同步到 Svelte store。`Actions.svelte` 中的“候选”和“下一步”按钮只负责触发这些方法，不直接计算数独规则。这样的设计可以让提示逻辑集中在领域对象里，界面层只负责显示结果。

## 2. 你认为提示功能更属于 `Sudoku` 还是 `Game`？为什么？

我认为提示功能的核心更属于 `Sudoku`，但 `Game` 可以提供一层对外接口。

原因是提示功能本质上依赖数独规则。比如一个格子有哪些候选数，需要检查它所在的行、列、宫是否存在重复数字；下一步提示也需要基于当前棋盘计算每个空格的候选数。这些都属于数独盘面本身的规则，所以核心算法应该放在 `Sudoku` 中。

但是在实际游戏中，玩家可能处于普通模式，也可能处于探索模式。此时“当前棋盘”不一定是主线棋盘，也可能是探索分支棋盘。因此 `Game` 需要提供 `getCandidates()` 和 `getHint()` 这样的接口，用来判断当前应该从主线 Sudoku 还是探索 Sudoku 中获取提示。

所以我的设计是：

- `Sudoku` 负责真正的候选数和提示算法；
- `Game` 负责决定当前使用哪一个 Sudoku，并把提示结果提供给 UI；
- UI 只调用 `Game` 暴露的方法，不直接操作 `Sudoku` 内部数据。

这样既保持了规则逻辑的位置清晰，也能让提示功能适应普通模式和探索模式。

## 3. 你如何实现探索模式？

探索模式由 `Game` 对象负责实现。它的核心思想是：进入探索模式时，不直接修改主线棋盘，而是复制一份当前棋盘，在副本上进行尝试。

在代码中，普通模式下的当前棋盘保存在 `currentSudoku` 中。点击“开始探索”时，`Game.enterExplore()` 会执行 `currentSudoku.clone()`，生成一个新的 Sudoku 对象，并保存到 `explorationData.sudoku` 中。之后玩家的填数、撤销、重做都会作用在这个探索分支上，而不会直接影响主线棋盘。

探索模式有三个主要操作：

1. `enterExplore()`：复制当前主线棋盘，进入探索模式。
2. `commitExplore()`：如果探索分支没有冲突，就把探索分支合并回主线棋盘。
3. `discardExplore()`：丢弃探索分支，回到进入探索前的主线棋盘。

为了让普通模式和探索模式复用同一套填数、撤销、重做逻辑，我实现了类似 `getActiveState()` 的方法。它会判断当前是否处于探索模式：如果正在探索，就返回探索分支的 Sudoku、undo 栈和 redo 栈；如果没有探索，就返回主线 Sudoku、undo 栈和 redo 栈。这样 `guess()`、`undo()`、`redo()` 不需要写两套逻辑。

探索模式还会检查失败状态。如果探索分支出现数独冲突，`Game` 会记录该局面，并阻止提交探索结果。这样可以避免把错误分支正式合并进主线游戏。

## 4. 主局面与探索局面的关系是什么？

主局面是玩家正式游戏进度，探索局面是从主局面复制出来的临时分支。

在没有进入探索模式时，所有操作都直接作用在主局面上。主局面由 `Game` 中的 `currentSudoku` 保存，它代表当前正式的游戏状态。

当玩家进入探索模式时，程序会复制当前主局面，创建一个探索局面。探索局面和主局面在内容上起点相同，但之后互相独立。玩家在探索模式中填写数字、撤销、重做，只会改变探索局面，不会改变主局面。

如果玩家认为探索结果是正确的，可以点击“提交探索”。这时探索局面会替换主局面，成为新的正式游戏进度。如果玩家发现探索方向不对，可以点击“放弃探索”。这时探索局面会被丢弃，主局面保持进入探索前的状态。

因此二者关系可以理解为：

- 主局面是正式路线；
- 探索局面是从正式路线复制出来的试探路线；
- 提交探索会把试探路线变成正式路线；
- 放弃探索会删除试探路线，回到正式路线。

这种设计符合玩家解数独时的真实思路：不确定时先假设一个数字继续推理，发现矛盾就退回，确认可行再保留。

## 5. 你的 history 结构在本次作业中是否发生了变化？

发生了变化。

在 Homework 1 中，history 主要用于普通的撤销和重做。它只需要记录主线游戏中的操作历史。玩家每填一步，就保存一个旧状态到 undo 栈；撤销时从 undo 栈恢复；重做时从 redo 栈恢复。

在 Homework 2 中，因为增加了探索模式，history 不能只服务于主局面。探索分支也需要有自己的撤销和重做，否则玩家在探索时无法回退试错。因此我把 history 的概念扩展成了两套状态：

- 主局面的 `undoStackData` 和 `redoStackData`；
- 探索局面自己的 `explorationData.undoStack` 和 `explorationData.redoStack`。

这样普通模式和探索模式都有独立的历史记录。探索模式中的撤销不会影响主局面的撤销历史。只有当探索被提交时，才会把提交前的主局面保存进主线 undo 栈，使“提交探索”本身成为主线上的一次可撤销操作。

另外，本次作业还加入了失败局面记录。`Game` 使用 `failedBoardKeySet` 保存曾经失败过的探索盘面。这个结构不是传统意义上的 undo/redo history，但它也是一种历史信息，用来判断玩家是否又走到了之前已经失败的局面。

## 6. Homework 1 中的哪些设计，在 Homework 2 中暴露出了局限？

Homework 1 的设计可以支持基础数独游戏，但在 Homework 2 中暴露出几个局限。

第一个局限是 UI 和游戏逻辑耦合较强。如果页面组件或 store 直接修改二维数组，那么加入候选数、下一步提示、探索模式时，规则逻辑会分散到很多地方，不方便维护。

第二个局限是原来的状态结构只适合单一路线游戏。Homework 1 中玩家基本只有一条主线历史，撤销和重做也围绕这条主线进行。但是 Homework 2 需要支持探索分支，玩家可以在临时副本上试错，这就要求游戏状态能够区分主局面和探索局面。

第三个局限是提示功能需要更清楚的规则入口。候选数和下一步提示都需要频繁检查行、列、宫的约束。如果这些规则没有集中放在一个对象里，就容易出现重复代码，也不利于以后增加更复杂的提示策略。

第四个局限是固定格和玩家填写格需要更明确地区分。Homework 2 中有更多操作会修改盘面，如果没有 `fixedMask` 这样的结构，程序就不容易稳定地判断哪些格子不能被修改。

因此 Homework 2 促使我把数独规则抽象成 `Sudoku`，把一局游戏流程抽象成 `Game`。这样比 Homework 1 中直接围绕页面状态写功能更清晰。

## 7. 如果重做一次 Homework 1，你会如何修改原设计？

如果重新做 Homework 1，我会一开始就把项目分成三层：领域层、状态连接层和 UI 层。

第一，我会提前设计 `Sudoku` 对象。它只负责数独盘面本身，例如保存棋盘、判断是否合法、计算候选数、复制棋盘、序列化和反序列化。这样后续加入提示功能时，不需要再从 UI 代码中拆规则。

第二，我会提前设计 `Game` 对象。它负责一局游戏的流程，例如玩家填数、撤销、重做、保存历史、判断完成。这样当 Homework 2 需要加入探索模式时，只需要在 `Game` 中扩展分支状态，而不是大规模修改页面组件。

第三，我会让 Svelte store 只做同步工作。store 不应该承担太多规则判断，而应该调用 `Game` 的方法，然后把结果同步给页面显示。这样 UI 层会更简单，也更容易测试。

第四，我会更早地使用快照和 clone。数独游戏里经常需要保存状态、恢复状态、尝试分支。如果一开始就保证 `getGrid()` 返回副本，`clone()` 能复制完整对象，就能减少外部代码误改内部状态的问题。

最后，我会在 Homework 1 中就写清楚对象职责和接口。例如哪些方法属于 `Sudoku`，哪些方法属于 `Game`，UI 可以调用哪些方法。这样 Homework 2 的扩展会更自然，代码结构也更稳定。
